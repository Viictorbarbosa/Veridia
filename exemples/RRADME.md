# Examples

This directory contains practical examples that demonstrate where Veridia is an appropriate architectural choice and where it is not.

The goal is not to showcase implementation details, but to illustrate the types of knowledge evolution problems Veridia is designed to solve.

## Purpose

These examples help readers understand:

- Which problems Veridia addresses.
- Which problems it does **not** address.
- The architectural trade-offs behind its design.
- How versioned, causality-aware memory differs from traditional retrieval approaches.

## Contents

| Document | Description |
|----------|-------------|
| `when-to-use.md` | Scenarios where Veridia is an appropriate solution. |
| `when-not-to-use.md` | Scenarios where alternative architectures are more suitable. |
| `policy-versioning.md` | Example of tracking evolving organizational policies. |
| `regulatory-compliance.md` | Example of retrieving knowledge across changing regulations. |
| `technical-documentation.md` | Example of managing multiple versions of technical documentation. |
| `research-history.md` | Example of representing evolving scientific knowledge over time. |

## Scope

The examples focus on architectural concepts rather than implementation.

They are intended to explain **why** Veridia is useful in specific scenarios, not how it is implemented internally.
